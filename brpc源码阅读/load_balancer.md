## brpc LoadBalancer 全景解析

brpc 提供了 7 种内置负载均衡算法，适用于不同的业务场景：

---

### 1. Round Robin (rr)

**原理**：轮询算法，按顺序依次选择列表中的下一台服务器，到达末尾后回到开头。

**工作流程**：
- 维护一个全局偏移量（offset），每次请求后递增
- 使用 `stride`（步长，通常为 1）计算下一个位置：`offset = (offset + stride) % n`
- 保证请求在服务器间绝对均匀分布

**适用场景**：
- 服务器配置、网络条件、负载完全同构的集群
- 需要请求绝对公平分配的场景

**特点**：
- 实现最简单，无状态
- 请求分布最均匀
- 不考虑服务器实际负载差异

---

### 2. Weighted Round Robin (wrr)

**原理**：加权轮询，根据服务器配置的权重值进行轮询，权重高的服务器获得更多请求。

**工作流程**：
- 服务器 tag 需配置为表示权重的 int32 数字（如 `tag="50"`）
- 使用平滑加权轮询算法（Smooth Weighted Round-Robin）
- 保证同一服务器被选中的结果较均衡地散开，避免连续命中同一高权重服务器

**适用场景**：
- 服务器性能异构的集群（如 4核8G 与 8核16G 混合部署）
- 需要按比例分配流量的场景

**特点**：
- 权重高的服务器获得更多流量，但分布平滑
- 相比简单轮询，能更好地利用异构资源

---

### 3. Random (random)

**原理**：从服务器列表中随机选择一台。

**工作流程**：
- 生成随机数，对服务器数量取模
- 直接返回对应索引的服务器

**适用场景**：
- 无状态服务
- 服务器完全同构且数量较多时
- 实现简单，无需维护状态

**特点**：
- 实现最简单，无状态
- 大样本下分布均匀，但短期可能有偏差
- 与 Round Robin 类似，不考虑实际负载

---

### 4. Weighted Random (wr)

**原理**：加权随机，根据服务器权重进行随机选择。

**工作流程**：
- 服务器 tag 配置权重值
- 根据权重比例生成随机选择区间
- 权重高的服务器有更大的被选中概率

**适用场景**：
- 异构集群，但不需要轮询的严格顺序
- 实现比 wrr 更简单，但分布不如 wrr 平滑

**特点**：
- 实现简单，纯随机
- 短期分布可能不均匀（方差较大）
- 适合权重差异不大的场景

---

### 5. Locality-Aware (la) ⭐ 重点

**原理**：局部感知负载均衡，**以下游节点的吞吐量（QPS）除以延迟（Latency）作为分流权值**。

**核心公式**：
```
Weight = QPS / Latency
```

**工作流程**：
1. **实时统计**：收集每个服务器的 QPS 和平均延迟
2. **权重计算**：`base_weight = (QPS * SCALE) / latency^p`（p 默认为 2）
3. **权重树查找**：使用完全二叉树（Weight Tree）实现 O(logN) 时间复杂度的选择
4. **动态调整**：
   - 延迟低的服务器获得更多流量，直到延迟升高
   - 使用 **Inflight Delay** 机制快速发现慢节点（追踪未完成 RPC 的耗时）
   - 错误请求会惩罚权重

**关键技术**：
- **DoublyBufferedData**：双缓冲数据结构，实现读写分离，读操作几乎无锁
- **Inflight Delay**：当未完成请求的平均延迟超过阈值时，提前惩罚权重，无需等待超时
- **权重树**：每个节点记录左子树权重和，支持高效权重选择

**适用场景**：
- 服务器性能差异大（异构集群）
- 网络条件不稳定
- 需要自动规避慢节点的场景
- 混合部署环境（如本机、本机架、跨机房）

**特点**：
- **自适应**：自动感知下游负载，动态调整
- **最优延迟**：理论上能达到全局最优的延迟分布
- **故障恢复**：快速规避故障节点，恢复后自动加回
- **复杂度高**：实现复杂，需要维护统计状态

---

### 6. Consistent Hashing (c_murmurhash / c_md5)

**原理**：一致性哈希，根据请求的 `request_code` 计算哈希值，映射到哈希环上的虚拟节点。

**工作流程**：
1. **构建哈希环**：
   - 每个真实服务器对应多个虚拟节点（默认 100-200 个）
   - 使用 MurmurHash64A 或 MD5 计算虚拟节点在环上的位置
2. **请求路由**：
   - 计算 `request_code` 的哈希值
   - 在环上顺时针查找最近的虚拟节点
   - 映射到对应的真实服务器
3. **节点变化**：
   - 节点下线：只影响该节点在环上的虚拟节点，其他请求路由不变
   - 节点上线：只影响新节点附近的请求

**适用场景**：
- **有状态服务**（如缓存系统，需要相同的 key 路由到相同的服务器）
- 需要会话保持（Session Stickiness）
- 服务器动态扩缩容频繁，需要最小化缓存失效

**特点**：
- **单调性**：节点增减只影响相邻节点，不会导致全局重新映射
- **平衡性**：虚拟节点机制确保负载均匀
- **分散性**：不同客户端视图下，同一请求映射到有限节点集
- **负载不均风险**：节点数量少时可能出现倾斜

---

### 7. 集群恢复限流（特殊模式）

**原理**：针对 "集群宕机后恢复" 场景的限流保护机制。

**工作流程**：
- 当集群中所有 server 都不可用时，进入恢复状态
- 假设能服务所有请求的 server 数量为 `min_working_instances`
- 当前可用 server 数量为 `q`
- 接受请求的概率为 `q / min_working_instances`，否则丢弃
- 若 `hold_seconds` 内 `q` 保持不变，则恢复全量流量

**适用场景**：
- 集群全部宕机后的渐进式恢复
- 避免瞬间流量压垮刚恢复的服务器

**配置方式**：
```
random:min_working_instances=6 hold_seconds=10
```

## 算法对比总结

| 算法 | 选择依据 | 适用场景 | 优点 | 缺点 |
|------|---------|---------|------|------|
| **rr** | 轮询顺序 | 同构集群 | 绝对公平，实现简单 | 不感知负载 |
| **wrr** | 权重+轮询 | 异构集群 | 按比例分配，分布平滑 | 不感知实时负载 |
| **random** | 随机 | 无状态服务 | 实现最简单 | 短期分布不均 |
| **wr** | 权重+随机 | 异构集群 | 实现简单 | 分布不如 wrr 平滑 |
| **la** ⭐ | QPS/Latency | 异构/不稳定网络 | 自适应，最优延迟 | 实现复杂 |
| **c_murmurhash** | 一致性哈希 | 有状态服务/缓存 | 节点变化影响小 | 可能负载不均 |
| **c_md5** | 一致性哈希（MD5） | 同 c_murmurhash | 哈希算法不同 | 同 c_murmurhash |

---

## 选择建议

| 场景 | 推荐算法 |
|------|---------|
| 服务器完全同构 | `rr` 或 `random` |
| 服务器性能差异大 | `wrr` 或 `la` |
| 需要会话保持/缓存 | `c_murmurhash` |
| 网络条件不稳定，需要自适应 | `la`（首选） |
| 简单快速部署 | `random` |
| 集群宕机恢复保护 | `rr:min_working_instances=N` |

---

## 关键实现机制

所有 LoadBalancer 都基于以下关键技术：

1. **DoublyBufferedData**：双缓冲数据结构，实现读写分离，读操作几乎无锁，写操作（服务器列表变更）不阻塞读

2. **健康检查**：自动检测故障节点，故障后定期探测，恢复后自动加回负载均衡池

3. **TLS（Thread Local Storage）**：每个线程维护独立的偏移量或状态，避免线程竞争

4. **Socket 复用**：长连接场景下，连接建立后复用，直到异常关闭
