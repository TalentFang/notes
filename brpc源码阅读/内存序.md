## 二、详解每层

### 第 1 层：缓存一致性（Cache Coherence）

**问题来源：多核各自有缓存**

```
CPU 0                    CPU 1                    内存
┌─────┐                 ┌─────┐                 ┌─────┐
│Cache│                 │Cache│                 │ x=0 │
│ x=0 │                 │ x=0 │                 └─────┘
└──┬──┘                 └──┬──┘
   │                       │
   ▼                       │
┌─────┐                    │
│ x=1 │  写入              │
└─────┘                    │
   │                       │
   │  缓存一致性协议         │
   │  (MESI)              │
   │──────────► 使 CPU1 的 x 失效
   │                       ▼
   │                    ┌─────┐
   │                    │ x=1 │  重新从内存加载
   │                    └─────┘
   ▼
┌─────┐
│ x=1 │  写回内存
└─────┘
```

**核心保证：**

| 特性 | 说明 |
|:---|:---|
| **写传播** | 一个核的写入，最终所有核都能看到 |
| **写串行化** | 所有核对同一地址的写入顺序达成一致 |
| **自动维护** | 硬件通过 MESI 等协议自动处理，程序员无感知 |

**关键局限：**

```
缓存一致性只保证：同一地址的读写顺序
不保证：不同地址之间的顺序！

示例：
CPU 0:              CPU 1:
x = 1;              if (y == 1)
y = 1;                  assert(x == 1);  // 可能失败！

问题：y=1 可能先于 x=1 被 CPU 1 看到
原因：缓存一致性不控制多变量顺序
```

### 第 2 层：内存一致性（Memory Consistency）

**问题：多变量操作的可见顺序**

```
程序员期望的顺序（程序序）：
─────────────────────────
x = 1;      // 操作 A
y = 1;      // 操作 B

期望：CPU 1 看到 y=1 时，一定看到 x=1
即：A 在 B 之前完成（A → B）

但硬件可能重排：
─────────────────────────
CPU 0 实际执行：
y = 1;      // B 先执行（更快，无依赖）
x = 1;      // A 后执行

结果：CPU 1 看到 y=1，但 x 还是 0！
```

**内存一致性模型：硬件提供的保证强度**

```
┌─────────────────────────────────────────────────────────┐
│              内存一致性模型谱系                          │
│                                                         │
│  顺序一致性（Sequential Consistency）                      │
│      ↓ "像单核一样，完全按程序序"                         │
│      实现：完全禁止重排，性能差                             │
│      代表：早期多核、SPARC TSO（接近）                      │
│                                                         │
│  处理器一致性（Processor Consistency）                     │
│      ↓ "每个核自己的写按序，但看到其他核的写可能乱"         │
│                                                         │
│  弱一致性（Weak Consistency）                              │
│      ↓ "只有同步操作（barrier）点才保证顺序"               │
│      代表：ARM、PowerPC                                  │
│                                                         │
│  释放一致性（Release Consistency）← C++ 默认              │
│      ↓ "用 acquire/release 标记同步点"                   │
│      代表：现代 x86/ARM、C++11 内存模型                    │
└─────────────────────────────────────────────────────────┘
```

**x86 vs ARM 对比：**

```
x86（强模型）：
─────────────────────────
- 自动保证：Store-Store 顺序、Load-Load 顺序
- 不允许：写读重排（Write-Read reordering）
- 程序员：很少需要 barrier，relaxed 操作安全

ARM（弱模型）：
─────────────────────────
- 允许：任意重排（只要单核语义正确）
- 程序员：必须用 barrier 保证多核顺序
- 性能：更高（重排优化空间大），但编程复杂
```

### 第 3 层：内存序（Memory Ordering）

**程序员控制工具：C++11 标准定义**

```
┌─────────────────────────────────────────────────────────┐
│              C++ 内存序枚举                              │
│                                                         │
│  memory_order_relaxed ───────────────────────────────┐  │
│      ↓ "只保证原子性，无顺序"                          │  │
│      使用：纯计数器，与其他变量无关联                    │  │
│                                                         │
│  memory_order_consume ───────────────────────────────┐  │
│      ↓ "数据依赖顺序"（极少用）                         │  │
│                                                         │
│  memory_order_acquire ─────────────────────────────┐  │
│      ↓ "读同步：本线程后续读/写不能重排到前面"           │  │
│      使用：读锁、检查标志位后访问数据                    │  │
│                                                         │
│  memory_order_release ─────────────────────────────┐  │
│      ↓ "写同步：本线程前面读/写不能重排到后面"           │  │
│      使用：写锁、发布数据后设置标志位                    │  │
│                                                         │
│  memory_order_acq_rel ───────────────────────────┐  │
│      ↓ "读写同步：同时有 acquire 和 release 语义"       │  │
│      使用：RMW（Read-Modify-Write）操作                 │  │
│                                                         │
│  memory_order_seq_cst ─────────────────────────┐  │
│      ↓ "全局顺序：所有线程看到一致的操作顺序"            │  │
│      使用：最强保证，但性能最差                           │  │
└─────────────────────────────────────────────────────────┘
```

## 三、三者关系图解

```
┌─────────────────────────────────────────────────────────┐
│                    完整数据流                            │
│                                                         │
│  程序员代码                                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │  std::atomic<int> flag{0};                      │    │
│  │  int data = 0;                                  │    │
│  │                                                 │    │
│  │  // 线程 A                                       │    │
│  │  data = 42;          // 普通写                    │    │
│  │  flag.store(1, memory_order_release);  // 标记   │    │
│  │                                                 │    │
│  │  // 线程 B                                       │    │
│  │  if (flag.load(memory_order_acquire) == 1) {    │    │
│  │      assert(data == 42);  // 保证看到           │    │
│  │  }                                              │    │
│  └─────────────────────────────────────────────────┘    │
│                      │                                   │
│                      ▼                                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │  内存序（程序员显式控制）                          │    │
│  │  - release：保证 data=42 在 flag=1 之前完成       │    │
│  │  - acquire：保证看到 flag=1 后，能看到 data=42      │    │
│  │  - 组合：建立 A→B 的 happens-before 关系           │    │
│  └─────────────────────────────────────────────────┘    │
│                      │                                   │
│                      ▼                                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │  内存一致性（硬件提供的保证）                        │    │
│  │  - x86：release/acquire 只需编译器屏障             │    │
│  │  - ARM：需要 dmb（数据内存屏障）指令                 │    │
│  │  - 硬件确保屏障前后的操作不重排                      │    │
│  └─────────────────────────────────────────────────┘    │
│                      │                                   │
│                      ▼                                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │  缓存一致性（硬件自动维护）                          │    │
│  │  - MESI 协议确保 flag 和 data 的最新值传播          │    │
│  │  - 写传播：A 的写入最终到达 B 的缓存               │    │
│  │  - 写串行化：所有核看到 flag 和 data 的修改顺序一致  │    │
│  └─────────────────────────────────────────────────┘    │
│                      │                                   │
│                      ▼                                   │
│              [物理硬件执行]                               │
└─────────────────────────────────────────────────────────┘
```

## 四、实例分析：为什么需要三层？

### 场景：简单的标志位同步

```
代码：
─────────────────────────
std::atomic<bool> ready{false};
int data = 0;

// 线程 A（生产者）
void produce() {
    data = 42;                          // ① 普通写
    ready.store(true, std::memory_order_release);  // ② 标记
}

// 线程 B（消费者）
void consume() {
    while (!ready.load(std::memory_order_acquire));  // ③ 等待
    assert(data == 42);                 // ④ 使用数据
}

问题：没有 release/acquire 会怎样？
─────────────────────────

情况 1：只有缓存一致性
- ① 和 ② 可能乱序到达 CPU B
- B 看到 ready=true，但 data 还是 0
- assert 失败！

情况 2：加上 release/acquire（内存序）
- release：① 必须在 ② 之前完成（编译器+硬件屏障）
- acquire：④ 必须在 ③ 之后执行，且看到 ① 的结果
- 建立 happens-before：① → ② → ③ → ④
- assert 成功！

情况 3：x86 vs ARM 实现差异
- x86：release/acquire 只需编译器屏障（硬件强模型）
- ARM：需要 dmb ishld（加载屏障）和 dmb ishst（存储屏障）
```

## 五、brpc 中的实际体现

```cpp
// 1. call_id() - 利用 x86 强模型，用 relaxed
butil::atomic<uint64_t>* target = ...;
uint64_t loaded = target->load(butil::memory_order_relaxed);
// x86 上就是普通 mov，因为：
// - 单变量操作
// - 无其他数据依赖
// - 后续 CAS 提供完整同步

// 2. 引用计数 - 需要 acq_rel 保证生命周期
butil::atomic<int32_t> _ref{1};
bool last = _ref.fetch_sub(1, butil::memory_order_acq_rel) == 1;
// acq_rel：保证之前的操作完成，后续操作看到最新值

// 3. 数据发布 - 经典 release-acquire
void publish(Data* d) {
    data.store(d, std::memory_order_relaxed);
    ready.store(true, std::memory_order_release);  // 发布
}

Data* consume() {
    while (!ready.load(std::memory_order_acquire));  // 获取
    return data.load(std::memory_order_relaxed);     // 保证看到完整 Data
}

// 4. 全局顺序 - 极少使用 seq_cst
std::atomic<int> seq{0};
seq.store(1, std::memory_order_seq_cst);  // 所有线程看到一致顺序
```

## 六、一句话总结三层关系

| 层次 | 控制者 | 解决的问题 | 一句话 |
|:---|:---|:---|:---|
| **缓存一致性** | 硬件 | 同一地址的最新值 | "多核缓存同步，自动透明" |
| **内存一致性** | 硬件+程序员 | 多变量操作的可见顺序 | "硬件提供模型，程序员选择强度" |
| **内存序** | 程序员 | 显式控制操作顺序 | "用原子操作和屏障，建立 happens-before" |

> **缓存一致性是基础（硬件搞定），内存一致性是选项（硬件提供、程序员选择），内存序是工具（程序员显式控制）。三者协同，构建正确的并发程序。**
