## 1. 高级负载均衡策略详解

### 1.1 LocalityAware（局部性感知负载均衡）

#### 核心思想

LocalityAware 是一种**自适应、反馈驱动的智能负载均衡策略**。它的设计理念是：**根据节点的实时表现动态调整流量分配，优先选择"表现更好"的节点**。

#### 评估维度

| 维度 | 说明 | 影响 |
|------|------|------|
| **延迟（Latency）** | 请求响应时间 | 延迟越高，得分越低 |
| **错误率（Error Rate）** | 失败请求占比 | 错误越多，得分越低 |
| **并发度（Concurrency）** | 当前处理中的请求数 | 负载越高，得分越低 |
| **权重（Weight）** | 服务端声明的容量 | 权重越高，基础得分越高 |

#### 工作原理

```
流量分配决策流程：

1. 收集统计信息
   ├── 每个节点维护一个滑动窗口统计器
   ├── 记录最近 N 次请求的延迟和结果
   └── 计算 P99 延迟、平均错误率

2. 计算节点得分
   Score = (Weight × LatencyFactor × ErrorFactor) / (Concurrency + 1)
   
   其中：
   - LatencyFactor = 1 / (1 + P99Latency_ms)  [延迟越低，因子越大]
   - ErrorFactor = (1 - ErrorRate)^2           [错误率影响非线性放大]

3. 选择节点
   ├── 按得分排序所有节点
   ├── 引入随机扰动避免热点（Jitter）
   └── 选择得分最高的节点

4. 反馈闭环
   └── 每次请求完成后更新统计，影响下次决策
```

#### 优势与适用场景

| 优势 | 说明 |
|------|------|
| **自动容错** | 故障节点自动被降权，无需人工干预 |
| **延迟优化** | 慢节点自动减少流量，降低整体 P99 |
| **异构环境适配** | 机器配置不同时，自动按能力分配流量 |
| **渐进式恢复** | 故障恢复后，流量逐步回流（非瞬间切换） |

**适用场景**：大规模集群、节点性能差异大、网络环境复杂的情况。

### 1.2 ConsistentHash（一致性哈希）

#### 核心思想

一致性哈希解决的是**有状态服务**或**缓存场景**下的路由问题：**相同的请求 key 总是路由到相同的节点**，即使节点数量变化，也只有少量 key 需要重新映射。

#### 哈希环构建

```
哈希空间：0 到 2^32-1（或 2^64-1）

物理节点映射：
┌─────────────────────────────────────────────────────────┐
│                    哈希环（顺时针）                       │
│                                                          │
│    0                                                      │
│    ├───────────────────────────────────────────────┐     │
│    │                                               │     │
│    │   Node-A#0 ──▶ 哈希值: 0x12345678            │     │
│    │   Node-A#1 ──▶ 哈希值: 0x23456789  (虚拟节点)  │     │
│    │   Node-A#2 ──▶ 哈希值: 0x3456789A  (虚拟节点)  │     │
│    │                                               │     │
│    │   Node-B#0 ──▶ 哈希值: 0x87654321            │     │
│    │   Node-B#1 ──▶ 哈希值: 0x98765432            │     │
│    │   Node-B#2 ──▶ 哈希值: 0xA9876543            │     │
│    │                                               │     │
│    │   Node-C#0 ──▶ 哈希值: 0xABCDEF00            │     │
│    │   ...                                         │     │
│    │                                               │     │
│    └───────────────────────────────────────────────┘     │
│    2^32-1                                                 │
└─────────────────────────────────────────────────────────┘

虚拟节点（Virtual Nodes）：
- 每个物理节点映射为 100-200 个虚拟节点
- 目的：均衡分布，避免数据倾斜
```

#### 请求路由

```
请求 key："user_12345"

1. 计算 hash("user_12345") = 0x50000000

2. 在哈希环上顺时针查找第一个 >= 0x50000000 的节点
   └── 找到 Node-B#1 (0x98765432)

3. 请求路由到 Node-B

4. 如果 Node-B 故障：
   └── 顺时针找到下一个节点 Node-C#0
   └── 只有 Node-B 上的 key 需要迁移
```

#### 节点增删的影响

| 操作 | 影响范围 | 说明 |
|------|---------|------|
| **添加节点** | 仅相邻节点的部分 key | 新节点分担相邻节点的负载 |
| **删除节点** | 仅该节点的 key | 全部迁移到顺时针下一个节点 |

**优势**：相比取模哈希（hash % N），节点变化时**迁移成本极低**（1/N 的数据）。

#### 适用场景

| 场景 | 原因 |
|------|------|
| **分布式缓存**（如 Memcached） | 相同 key 必须命中相同节点 |
| **有状态服务**（如会话保持） | 用户会话固定在特定节点 |
| **分片数据库** | 数据分片路由，避免跨节点查询 |
| **流式计算** | 相同用户/订单的数据路由到相同处理节点 |
