# 1. 知识库和直接将知识作为prompt输入给模型有什么区别？

## 一、核心区别一句话

| 维度       | 知识库（RAG）       | Prompt内嵌知识                   |
| -------- | -------------- | ---------------------------- |
| **本质**   | 外挂式检索，按需取用     | 内嵌式输入，全量加载                   |
| **类比**   | 去图书馆查资料，只借需要的书 | 把所有书搬回家，一次全堆在桌上              |
| **知识容量** | 无上限（GB/TB级）    | 受模型上下文窗口限制（通常4K-200K tokens） |
| **成本**   | 低（仅检索相关片段）     | 高（每次请求都消耗大量token）            |

## 二、技术原理对比

### 2.1 Prompt内嵌知识：全量注入模式

```
用户提问 → 拼接完整Prompt（含全部知识） → 发送给LLM → 获得回答
```

**特点**：
- 所有知识一次性全部放入上下文
- LLM在推理时能看到所有信息
- 适合小规模、高关联度的知识

**问题**：
1. 如果资料有1万行代码，token消耗巨大
2. 超出上下文窗口就无法处理
3. 即使没超出，大量无关信息也会干扰LLM注意力

### 2.2 知识库（RAG）：按需检索模式

```
用户提问 → 向量检索 → 召回相关片段 → 拼接到Prompt → 发送给LLM → 获得回答
          ↓
      知识库（向量数据库）
     （存储全量源码片段）
```

**特点**：
- 只检索最相关的3-5个片段
- 知识总量无上限（可存储整个Linux内核源码）
- 每次消耗token远小于全量注入

**示例流程**：
```
用户问："Producer的发送逻辑是什么？"

知识库检索：
→ 召回 "KafkaProducer.java" 中 send() 方法的代码片段
→ 召回 "RecordAccumulator.java" 中 addRecord() 的代码片段
→ 召回 "Sender.java" 中 run() 的代码片段

拼接到Prompt：
[仅上述3个片段，共500行代码]

回答：基于这些片段分析Producer的发送流程...
```

## 三、最佳实践：两者结合使用

在实际的代码分析Agent中，最有效的策略是**Prompt定行为，知识库供素材**：

```yaml
# Prompt部分（定义行为）
role: 代码架构师
task: 分析用户指定的代码模块
style: 生成Mermaid架构图，带模块说明
constraints: 如果检索不到相关信息，明确告知用户

# 知识库部分（提供素材）
knowledge_base:
  - 全量源码库（用于精准检索）
  - 技术文档库（用于补充说明）
  - 历史分析报告（用于参考上下文）
```

### 工作流程

```
用户提问
    ↓
Prompt定义AI角色、任务、输出格式
    ↓
知识库检索相关代码片段
    ↓
组合：Prompt + 检索结果
    ↓
LLM生成答案（受Prompt约束，基于检索结果）
    ↓
输出：带引用来源的架构图/分析报告
```
