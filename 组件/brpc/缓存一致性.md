 缓存一致性（Cache Coherence）是多核 CPU 架构中的核心问题，指**多个 CPU 核心各自的缓存中，同一份内存数据如何保持同步**。

---

## 问题起源：为什么需要缓存一致性？

```
┌─────────┐         ┌─────────┐
│  CPU 0  │         │  CPU 1  │
│ ┌─────┐ │         │ ┌─────┐ │
│ │Cache│ │         │ │Cache│ │
│ │ x=1 │ │         │ │ x=0 │ │  ← 同一变量，值不同！
│ └─────┘ │         │ └─────┘ │
└────┬────┘         └────┬────┘
     │                   │
     └───────┬───────────┘
             ▼
        ┌─────────┐
        │ 内存 x=0 │  ← 主内存
        └─────────┘

场景：
1. CPU 0 读取 x，缓存为 1（并修改）
2. CPU 1 读取 x，缓存为 0（旧值）
3. 两个核心看到不同的值 → 程序逻辑错误！
```

---

## 缓存一致性的定义

> **缓存一致性协议**确保：任何时刻，**所有 CPU 核心对同一内存地址的读取，都能看到最新的写入值**。

关键性质（MESI 协议的核心）：

| 性质 | 说明 |
|------|------|
| **写传播（Write Propagation）** | 一个核心的写入，必须传播到其他核心 |
| **写串行化（Write Serialization）** | 所有核心看到的写入顺序一致 |

---

## 主流解决方案：MESI 协议

MESI 是 Intel、AMD 等 x86 CPU 使用的缓存一致性协议，每个缓存行有 4 种状态：

| 状态 | 名称 | 含义 |
|:--:|:---|:---|
| **M** | Modified（修改） | 独占且已修改，与内存不同，写回后变为 E |
| **E** | Exclusive（独占） | 独占且未修改，可直接写无需通知总线 |
| **S** | Shared（共享） | 多核心共享，只读，写之前需广播失效 |
| **I** | Invalid（失效） | 无效，读取时需从其他缓存或内存加载 |

### 状态转换示例

```
CPU 0 写入 x（状态变化流程）：

初始：CPU0(I), CPU1(I), 内存(x=0)

1. CPU0 读取 x
   → 从内存加载，CPU0(E), CPU1(I), 内存(x=0)
   
2. CPU1 读取 x  
   → 从 CPU0 嗅探到，CPU0(S), CPU1(S), 内存(x=0)
   
3. CPU0 写入 x=1
   → 广播 Invalidate，CPU0(M), CPU1(I), 内存(x=0)
   
4. CPU1 读取 x
   → 从 CPU0 获取最新值，CPU0(S), CPU1(S), 内存(x=0)
   → 稍后写回内存
```

## 硬件实现机制

### 1. 总线嗅探（Bus Snooping）

```
┌─────────┐     ┌─────────┐
│  CPU 0  │◄───►│  CPU 1  │
│  Cache  │     │  Cache  │
└────┬────┘     └────┬────┘
     │               │
     └───────┬───────┘
             ▼
        ┌─────────┐
        │  总线   │ ◄── 所有核心监听总线事务
        │(Bus)    │     发现相关地址则更新状态
        └─────────┘
```

### 2. 目录协议（Directory-based）

用于 NUMA 架构（多路服务器），通过中央目录跟踪缓存行位置，减少广播开销。

## 对程序员的影响

### 1. 可见性问题（Visibility）

```cpp
// 线程 1 (CPU 0)
data = 42;          // 写入缓存，未刷回内存
ready = true;       // 另一个缓存行

// 线程 2 (CPU 1)  
while (!ready);     // 可能永远看不到 ready=true！
assert(data == 42); // 可能看到 data=0（旧值）
```

**原因：** 
- `data` 和 `ready` 在不同缓存行
- 写入顺序在 CPU 0 的 Store Buffer 中重排
- CPU 1 的缓存未失效，看到旧值

### 2. 内存序（Memory Ordering）

```cpp
// C++ 原子操作解决可见性和顺序问题
std::atomic<int> data{0};
std::atomic<bool> ready{false};

// 线程 1
data.store(42, std::memory_order_relaxed);
ready.store(true, std::memory_order_release);  // 释放语义：之前的写不能重排到后面

// 线程 2
while (!ready.load(std::memory_order_acquire)); // 获取语义：之后的读不能重排到前面
assert(data.load(std::memory_order_relaxed) == 42);  // 保证看到 42
```

### 3. 伪共享（False Sharing）

```cpp
struct {
    std::atomic<int> a;  // 缓存行 0 (64字节)
    std::atomic<int> b;  // 同一缓存行！
} vars;

// CPU 0 不断写 a，CPU 1 不断写 b
// 导致缓存行在两个核心间来回弹跳（Ping-Pong），性能暴跌
```

**解决：** 使用 `alignas(64)` 将变量分到不同缓存行。

## 缓存一致性 vs 内存一致性

| 概念 | 层级 | 解决的问题 |
|:---|:---|:---|
| **缓存一致性** | 硬件层 | 多核缓存同步，保证**单变量**的最新值 |
| **内存一致性** | 软硬件协同 | 多变量操作的**全局顺序**，程序员看到的执行顺序 |

```
缓存一致性保证：
CPU0: x=1 ───────────────► 所有核心最终看到 x=1

内存一致性保证：
CPU0: x=1; y=2;  ────────► 其他核心看到 x=1 在 y=2 之前（不颠倒）
```

## 总结

```
┌─────────────────────────────────────────┐
│           缓存一致性协议 (MESI)          │
│  ┌─────────┐    总线嗅探    ┌─────────┐ │
│  │ CPU缓存 │ ◄────────────► │ CPU缓存 │ │
│  │  (M/E/S/I)              │  (M/E/S/I)│ │
│  └────┬────┘               └────┬────┘ │
│       └───────────┬─────────────┘       │
│                   ▼                     │
│              ┌─────────┐                │
│              │ 主 内 存 │                │
│              └─────────┘                │
└─────────────────────────────────────────┘
        ▲
        │ 需要配合
        ▼
┌─────────────────────────────────────────┐
│      内存屏障 / 原子操作 (软件层)         │
│   std::atomic, memory_order_xxx        │
│   解决：可见性 + 指令重排 + 顺序保证      │
└─────────────────────────────────────────┘
```

**一句话：** 缓存一致性是硬件自动维护的**单变量正确性**，程序员仍需关注**多变量顺序**和**可见性**，通过原子操作和正确的内存序与硬件协同工作。
