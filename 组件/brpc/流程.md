## brpc 整体层次架构

brpc 采用经典的分层设计，主要包括以下几个层次：

```
┌─────────────────────────────────────────────────────────┐
│                    应用层 (Service Layer)                 │
│         (Service, Controller, Closure, Protobuf)          │
├─────────────────────────────────────────────────────────┤
│                    协议层 (Protocol Layer)                │
│    (baidu_std, hulu_pbrpc, sofa_pbrpc, thrift, http...) │
├─────────────────────────────────────────────────────────┤
│                    序列化层 (Serialization)               │
│         (Protobuf, JSON, Thrift, FlatBuffers...)          │
├─────────────────────────────────────────────────────────┤
│                    网络传输层 (Transport Layer)           │
│         (Socket, Connection, EventDispatcher, IOBuf)      │
├─────────────────────────────────────────────────────────┤
│                    基础组件层 (Infrastructure)            │
│    (bthread, bvar, brpc::ResourcePool, Timer...)          │
└─────────────────────────────────────────────────────────┘
```

### 各层功能详解

| 层次 | 核心功能 |
|:---|:---|
| **应用层** | 提供 `brpc::Server` 和 `brpc::Channel`，封装 RPC 调用接口，管理服务生命周期 |
| **协议层** | 定义消息格式，支持多种协议（内置 baidu_std、HTTP、Thrift 等），可扩展自定义协议 |
| **序列化层** | 负责请求/响应消息的序列化与反序列化，默认 Protobuf，支持多种格式 |
| **网络传输层** | 管理 Socket 连接、数据收发、流量控制、连接池，是 brpc 的核心 |
| **基础组件层** | `bthread`（M:N 线程模型）、`bvar`（监控统计）、内存池、定时器等基础设施 |

## 1. 客户端请求发起

### 1.1 用户调用Stub方法

用户通常通过Protocol Buffers生成的Stub类调用服务方法：

```cpp
// 创建请求和响应对象
test::EchoRequest request;
test::EchoResponse response;

// 设置请求参数
request.set_message("hello world");

// 创建控制器
brpc::Controller cntl;

// 调用服务方法（同步调用）
stub.Echo(&cntl, &request, &response, NULL);

// 检查调用结果
if (!cntl.Failed()) {
    std::cout << "Response: " << response.message() << std::endl;
}
```

### 1.2 Stub调用Channel::CallMethod

## 2. Channel处理请求

### 2.1 CallMethod初始化

`Channel::CallMethod`是请求处理的核心入口：

```cpp
void Channel::CallMethod(const google::protobuf::MethodDescriptor* method,
                         google::protobuf::RpcController* controller_base,
                         const google::protobuf::Message* request,
                         google::protobuf::Message* response,
                         google::protobuf::Closure* done) {
    const int64_t start_send_real_us = butil::gettimeofday_us();
    Controller* cntl = static_cast<Controller*>(controller_base);
    
    // 初始化Controller
    cntl->OnRPCBegin(start_send_real_us);
    cntl->_request_protocol = _options.protocol;
    cntl->_preferred_index = _preferred_index;
    cntl->_retry_policy = _options.retry_policy;
    cntl->_pack_request = _pack_request;
    cntl->_method = method;
    cntl->_auth = _options.auth;
    
    // ...
}
```


### 2.3 选择服务器

根据连接类型和负载均衡策略选择服务器：

- **单服务器模式**：直接使用预连接的Socket
- **多服务器模式**：通过LoadBalancer选择服务器

```cpp
if (SingleServer()) {
    cntl->_single_server_id = _server_id;
    cntl->_remote_side = _server_address;
} else {
    cntl->_lb = _lb;  // 负载均衡器
}
```

### 6.3 调用回调

- **同步调用**：唤醒等待的线程
- **异步调用**：调用用户提供的done回调函数

```cpp

    done->Run();  // 异步调用

    bthread_id_unlock(correlation_id);  // 同步调用，唤醒等待线程
}
```

## 8. 关键组件与类

### 8.1 Channel

- **核心通信通道**：负责与服务器建立和管理连接
- **负载均衡**：支持多种负载均衡策略
- **连接管理**：支持单连接、连接池和短连接模式

### 8.2 Controller

- **RPC控制器**：控制RPC调用过程
- **状态管理**：记录调用状态、错误信息等
- **超时控制**：支持设置和检查超时

### 8.3 InputMessenger

- **消息输入处理**：负责接收和解析网络数据
- **协议处理**：支持多种协议的消息解析
- **异步调度**：将消息处理任务调度到bthread执行

### 8.4 Socket

- **网络连接抽象**：封装底层网络连接
- **数据传输**：提供读写数据的接口
- **事件处理**：支持事件驱动的网络编程

## 9. 总结

brpc从发起请求到返回响应的流程是一个**高度模块化**、**异步高效**的过程，主要包括：

1. **客户端发起请求**：通过Stub调用Channel的CallMethod方法
2. **Channel处理请求**：初始化、序列化、选择服务器、发送请求
3. **服务器接收处理**：InputMessenger接收、解析、调用服务方法
4. **服务器返回响应**：序列化、发送响应
5. **客户端接收响应**：解析、匹配请求、调用回调

这种设计确保了brpc在高并发场景下的高性能和可靠性，同时提供了简单易用的API接口。
