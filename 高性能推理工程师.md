## 二、针对 RPC 背景的定制化学习路线

### 阶段一：建立 AI 推理的核心认知（2 周）

**目标**：理解 LLM 推理的特殊性，以及与 RPC 框架的关联点。

**学习内容**：

2. **推理瓶颈**：显存带宽、KV cache 管理、请求调度
3. **与 RPC 的关联**：
   - 推理请求的调度与 RPC 的请求调度有何异同
   - 流式推理如何映射到流式 RPC 模型
   - 为什么需要 continuous batching（类比协程调度）

**产出**：一篇笔记，梳理“推理请求调度 vs brpc 调度”的对比

### 阶段二：用 RPC 视角深入推理引擎（4 周）

**目标**：阅读 vLLM 源码，重点理解其调度设计，并与 brpc 对比。

#### 2.1 vLLM 源码阅读重点

| vLLM 模块 | 文件路径 | 与你 RPC 经验的关联 |
|----------|---------|-------------------|
| 请求调度 | `vllm/core/scheduler.py` | 类比 brpc 的请求调度策略 |
| 批处理管理 | `vllm/core/block_manager.py` | 类比内存池设计，但 GPU 显存管理更复杂 |
| 异步执行 | `vllm/engine/async_llm_engine.py` | 类比 Reactor + 线程池模型 |
| 流式输出 | `vllm/entrypoints/api_server.py` | 类比流式 RPC 的实现 |

#### 2.2 深度对比分析

建议做一份对比文档，例如：

```
【对比分析：vLLM Scheduler vs brpc bthread】

1. 设计目标
   - brpc：让 CPU 密集型任务也能享受协程的轻量调度
   - vLLM：让 GPU 推理请求通过 continuous batching 提高吞吐

2. 调度时机
   - brpc：bthread 在用户态切换，遇到阻塞时 yield
   - vLLM：在每个 decode step 后重新调度，决定下一批执行哪些请求

3. 资源管理
   - brpc：bthread 的栈内存池管理
   - vLLM：PagedAttention 的 KV cache 显存管理（思想上有异曲同工之妙）

4. 我可以复用的经验
   - brpc 的 bthread 让我理解了“用户态调度”的价值
   - 这个思维可以直接迁移到理解 continuous batching
```

### 阶段三：用 C++ 实现一个最小推理调度器（3 周）

**目标**：将 RPC 经验转化为可展示的代码，体现你的工程能力。

**项目建议**：用 C++ 实现一个 **极简推理调度器**，核心功能：
- 模拟多个推理请求（每个请求有 input_len, output_len）
- 实现 continuous batching 调度逻辑
- 对比 static batching vs continuous batching 的吞吐差异

**技术选型**：
- 可以用 CPU 模拟 GPU 计算（sleep 模拟推理时间）
- 重点展示调度器的设计，而非模型推理本身

**代码结构**（借鉴 brpc 风格）：
```cpp
// 借鉴 brpc 的设计思想
class Scheduler {
public:
    void AddRequest(Request req);
    void Run();
    
private:
    void Schedule();           // 每轮决定执行哪些请求
    void ExecuteBatch();       // 执行一个 batch
    void UpdateStatus();       // 更新进行中的请求
    
    std::vector<Request> waiting_;
    std::vector<RunningRequest> running_;
    int max_batch_size_;
    int max_wait_time_;
};
```

**这个项目的好处**：
- 展示你对调度机制的理解
- 体现 C++ 工程能力
- 面试时可以很自然地引出 brpc 经验

## 三、面试准备建议

### 可以重点准备的差异化问题

**1. “你如何设计一个高并发的推理服务？”**
> 你可以回答：结合 muduo 的 Reactor 模型处理网络 I/O，借鉴 brpc 的 bthread 思想处理请求调度，用独立线程池管理 GPU 计算，避免阻塞主循环。同时参考 vLLM 的 continuous batching 在 GPU 侧做进一步优化。

**2. “brpc 的 bthread 和 vLLM 的 continuous batching 有什么异同？”**
> 这是你最能发挥优势的问题。可以回答：两者都是用户态调度，但 bthread 解决 CPU 等待问题，continuous batching 解决 GPU 利用率问题。bthread 在阻塞时 yield，continuous batching 在每次 decode 后重新调度。PagedAttention 的显存管理与 bthread 的栈内存池设计有相通之处。

**3. “你读过 vLLM 源码吗？有什么设计上的见解？”**
> 结合你的 brpc 经验，从调度、内存管理、流式 RPC 三个角度分析，展示深度思考。

**4. “你之前做 RPC 框架，现在转 AI 推理，有什么优势？”**
> 你可以强调：RPC 框架让我深刻理解高并发服务的核心挑战——调度、资源管理、网络 I/O，这些正是推理服务的核心。而且现在推理框架的演进方向（如 continuous batching），与 RPC 框架的演进思路异曲同工。
